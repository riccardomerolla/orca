# Renderer rework + interactive-mode plan

Temporary working doc. Tracks the two threads of work the user requested.

---

## Part 1 — Terminal renderer rework

### Problem statement

The names lie about what each class does, and `out: PrintStream` ends up
shared across three threads with a `synchronized` block holding things
together. The user wants:

- One class that *owns* `out`. Its public interface is "display message"
  / "advance status". The background thread (animator) is an
  implementation detail.
- Internal serialisation via an Ox actor — not a `lock.synchronized`.
- That class is used by the listener path AND by the conversation
  renderer (helper class used inside `drive`).

### Research summary (from research agent)

Today there are three writers to `out`:

- **A** TerminalInteraction actor worker — `TerminalRendererState.onEvent` →
  `statusBar.appendLog` / `startStatus` / `stopStatus`.
- **B** Body thread (parked in `actor.ask` for drive) — runs
  `driveConversation` → `TerminalConversationRenderer` → `statusBar.*`.
- **C** StatusBar's animator forkDiscard — `statusBar.tick`.

`StatusBar`'s `synchronized` block coordinates A+B+C.

Why we can't put everything on one actor: `drive` parks the actor on a
blocking `conversation.events` iterator, so any tick routed through that
actor would freeze the spinner.

Misleading names:
- **StatusBar** does the whole event-log rendering, not just the status row.
  It IS the renderer.
- **TerminalRendererState** is an `OrcaListener` that does rendering, not
  just state. The actual state is `StageDepth` + `StageStack`.

### Target shape

Three classes with clean responsibilities:

```
TerminalOutput      // owns `out`, internal actor + animator; one writer thread
TerminalEventListener  // OrcaListener; holds stage state; tells TerminalOutput
ConversationRenderer   // per-conversation live renderer; tells TerminalOutput
```

`TerminalInteraction` thins down to: build the output + listener, expose
listeners and drive, route close.

Key design points:

1. **TerminalOutput owns its actor + animator + `out`.** No lock. Public
   methods are tells (`log`, `setStatus`, `clearStatus`), one ask
   (`close`) for shutdown drain. The animator is `forkDiscard` inside
   `start`; it `tell`s the renderer's own actor every `framePeriodMs`.
   Because rendering work is fire-and-forget tells, the renderer's actor
   is never parked — the spinner keeps ticking through `drive`.

2. **TerminalEventListener replaces TerminalRendererState.** Holds
   `StageStack`, `StageDepth`. `onEvent` formats and tells
   `TerminalOutput.log` / `setStatus`. May still run on its own actor
   (so `OrcaListener.onEvent` from multiple threads is serialised over
   stage state), OR be lock-free if listener dispatch already
   serialises — investigate before committing to a shape.

3. **ConversationRenderer (rename) keeps per-conversation state**
   (textBuffer, currentSection) but its writes go through
   `TerminalOutput.log` (tells). `promptApproval` uses
   `TerminalOutput.suspendStatus` (ask) so the bar clears before
   `readLine` and resumes after.

4. **Drive runs on caller's thread** (no `actor.ask`). Drive creates a
   `ConversationRenderer`, iterates events, telling `TerminalOutput`.
   When the conversation finishes, drive returns. Spinner ticks
   continue throughout because they hit the same `TerminalOutput` actor
   independently of the body thread.

5. **close()** asks `TerminalOutput` to clear the status; FIFO drain
   handles the pending tells.

### Tasks

- [ ] **R0.** Replace `Thread.sleep(framePeriodMs)` in
      `StatusBar.start` with `ox.sleep(framePeriodMs.millis)`. Trivial;
      do first to make the diff small.

- [ ] **R1.** `EventDispatcher` already documents that "concurrent
      `onEvent` calls fan out concurrently to each listener — listeners
      must themselves be thread-safe" (`EventDispatcher.scala:13-18`),
      so `TerminalEventListener` MUST serialise. Decision: use a
      simple `lock.synchronized` block around `onEvent`'s body. The
      listener's only output is fire-and-forget tells to
      `TerminalOutput`; tells are cheap, the lock is held for
      microseconds. An extra actor would be ceremony for the same
      effect.

- [ ] **R2.** Introduce `TerminalOutput` (new file) with internal
      actor + animator fork. Public API:
      - `log(text: String): Unit` — tell.
      - `setStatus(label: Option[String]): Unit` — tell.
      - `suspendStatus(): Unit` — ask. Clears the status row, sets a
        `paused` flag inside the actor; while paused, animator ticks
        short-circuit AND `log` tells are buffered into a queue
        instead of writing to `out`.
      - `resumeStatus(): Unit` — ask. Clears the flag, drains the
        buffer to `out` in arrival order, then resumes the animator.
      - `close(): Unit` — ask. Clears status and drains pending tells.
        Swallows throwables (matches today's `close()` semantics).
      Animator is a `forkDiscard` inside `start` that `ox.sleep`s,
      then `tell`s `self.tick()`. Animator runs in the same scope, so
      it's interrupted at scope-end.

- [ ] **R3.** Move stage state out of `TerminalRendererState` into
      `TerminalEventListener`. Keep `StageStack`/`StageDepth` as
      collaborators. The listener tells `TerminalOutput` for every
      effect; it does not call `out` directly. Use `lock.synchronized`
      per R1.

- [ ] **R4.** Rewire `TerminalConversationRenderer` →
      `ConversationRenderer`. It takes `TerminalOutput` in its
      constructor and tells it. Buffered-text + section state stay on
      the calling thread. `promptApproval` wraps `prompter.ask` in
      `output.suspendStatus()` … `output.resumeStatus()` in a
      `try/finally` so the buffer drains even if the user cancels.

- [ ] **R5.** Drop `actor.ask(_.driveConversation(...))` from
      `TerminalInteraction.drive`. Drive runs synchronously on the
      caller's thread, builds a `ConversationRenderer`, iterates
      events. Spinner stays live because `TerminalOutput` is a
      separate actor with its own worker.

- [ ] **R6.** `TerminalInteraction` thins down. It holds
      `TerminalOutput` + `TerminalEventListener`, exposes `listeners`
      (= `List(listener)`), forwards `drive` (build a
      `ConversationRenderer`, run on caller thread), and `close`
      (calls `output.close()`). No actor needed at this layer.

- [ ] **R7.** Tests, interleaved with each task as it lands. Specific
      additions:
      - **TerminalOutput**: spinner ticks continue while a long `log`
        handler is in flight (simulate via a fake `PrintStream` whose
        `flush` blocks on a latch).
      - **TerminalOutput**: `suspendStatus` buffers subsequent `log`
        tells, `resumeStatus` drains them in arrival order.
      - **Regression test for the spinner-during-drive bug**: scripted
        `Conversation` whose `events` iterator blocks on a latch;
        assert tick writes observed on the fake `PrintStream` during
        the block.
      - Update old tests (`StatusBarTest`, `TerminalInteractionTest`,
        `TerminalConversationRendererTest`) to point at the new class
        names. Tests that construct directly may need `supervised:`
        for the animator fork; OR offer a sync constructor that skips
        the fork (for unit tests that don't need animation).

- [ ] **R8.** Run the example to verify spinner spins through
      `Plan.interactive.from`.

## Part 2 — Interactive mode

### Problem statement

The user's hypothesis: "interactive" should mean the LLM can pause
mid-task to ask the user a free-form question and wait for the typed
answer. Today it doesn't — `interactive` only means "live streaming +
manual tool approval".

### Research summary (from research agent)

**Verdict: not implemented.**

- `design.md:64` and `README.md:118` describe the intended semantic:
  the agent can ask clarifying questions before producing output.
- `Conversation.sendUserMessage` exists in the trait but is a typed
  no-op on both backends (`ClaudeBackend.scala:78-101`,
  `CodexConversation.scala:69-73`). Both close stdin immediately
  after writing the prompt.
- `ConversationEvent` has eight variants — none represent
  "agent wants to ask the user".
- `TerminalConversationRenderer.promptApproval` is the only `readLine`
  in production code; it only handles `ApproveTool`.
- `UnapprovedPolicy.AskUser` is referenced nowhere except the enum
  definition — dead code today.
- `ClaudeBackend.scala:78-82` has a comment explicitly naming the gap
  ("AskUserQuestion tool calls. Future work").

So `Plan.interactive.from` today = autonomous + live event rendering +
tool-approval prompts. The agent cannot ask anything; the user can only
say "y" or "n" to a tool.

### Decision needed

Before implementation:

- **D1.** Is closing the gap in scope right now? It touches:
  - Claude backend (keep stdin open, plumb agent → user → agent rounds).
  - Codex backend (much harder — `codex exec` is one-shot; full
    feature needs a separate JSON-RPC server path; deferred in ADR 0007).
  - Trait `Interaction.drive` (no change if the event protocol is
    internal to `Conversation`; otherwise yes).
  - New `ConversationEvent` variant.
  - Renderer to display the question + read the answer.
  - Wiring `UnapprovedPolicy.AskUser` or removing it.
- **D2.** If yes, do we do Claude-only and document the codex gap, or
  hold off until both backends can do it?

### Implementation sketch (if approved)

1. Add `ConversationEvent.UserQuestion(text: String, respond: String => Unit)`.
2. Renderer: case for `UserQuestion` — clear status, print the question
   with a distinct glyph, call `prompter.ask`, invoke
   `respond(answer)`. Resume status afterwards.
3. Claude:
   - Stop closing stdin in `ClaudeBackend.openConversation`.
   - Expose an `ask_user` MCP tool. When the agent calls it, the
     conversation driver intercepts the tool call, emits
     `UserQuestion`, and feeds the answer back as a `tool_result`
     user turn.
   - `sendUserMessage` becomes a real write of a stream-json user turn.
4. Codex:
   - Defer. Document the asymmetry in the trait scaladoc.
   - Or: prompt-engineer the agent to surface a question via the
     structured output, then start a `codex exec resume` with the
     answer. Functional but ugly.
5. Decide `UnapprovedPolicy.AskUser`:
   - Wire it: interactive sessions show approval prompts regardless of
     `autoApprove`; autonomous denies.
   - Or remove it.

### Tasks

- [x] **I0.** Confirmed scope with user.

- [x] **I1.** Add `ConversationEvent.UserQuestion(text, respond)`.

- [x] **I2.** Render `UserQuestion` in `ConversationRenderer`.

- [x] **I4.** `Conversation.canAskUser` capability flag; document
      Codex asymmetry.

- [ ] **I5.** Wire or remove `UnapprovedPolicy.AskUser`.

### I3 reset: MCP path

Reality check: `claude --print --input-format stream-json` batches
stdin until EOF. The prompt-engineering [ASK_USER]: hack can't work
because the agent doesn't process turn-by-turn in that mode. The
Claude Agent SDK does support streaming input but it's Node/Python
only, not Scala. The Claude Code CLI itself has no streaming-input
flag (confirmed via docs).

**The right path is an MCP server.** Spawn a tiny HTTP MCP server
inside our Scala process that exposes an `ask_user` tool. Tell the
Claude CLI about it via `.mcp.json` (or `claude mcp add`). When the
agent needs a clarification, it calls `ask_user`; our server's
handler rendezvouses with the conversation thread, which emits
`UserQuestion` to the renderer, captures the typed answer, and
returns it to the MCP handler, which returns it to the agent.

Library: **softwaremill/chimp 0.1.8**. Built on Tapir-sync + Netty,
fits our Ox stack. `tool("ask_user").input[AskUserInput].handle(...)`
+ `mcpEndpoint(...)` + `NettySyncServer().port(0).start()`. Handler
is a pure `Input => Either[String, String]` running on a Netty
worker thread; it blocks while we round-trip through the host
process.

Module placement: new sub-package `orca.tools.claude.mcp` inside the
`claude` module. Nothing leaks outside the package — `ClaudeBackend`
is the only caller.

#### MCP sub-tasks

- [ ] **M1.** Add chimp dependency. Pin `com.softwaremill.chimp %
      core % 0.1.8` to the `claude` module. Verify it pulls Tapir
      1.13.x + Netty-sync; no foreign runtime.

- [ ] **M2.** Define `AskUserBridge` in
      `orca.tools.claude.mcp` — a tiny rendezvous abstraction. One
      side enqueues a question and blocks for the reply; the other
      side dequeues, handles, signals the reply. Two `ox.channels`
      `Channel`s (or one bidirectional dance) wrapped in a small
      class. Unit-test with one fork on each side.

- [ ] **M3.** Define `AskUserMcpServer` — wraps chimp's
      `mcpEndpoint` for the `ask_user` tool. Takes an
      `AskUserBridge`, starts Netty on port 0, exposes the bound
      URL. Lifecycle tied to the surrounding `supervised:` block via
      a fork. Smoke test: hit `/mcp` with a JSON-RPC
      `tools/list` request, assert `ask_user` is advertised.

- [ ] **M4.** `ClaudeBackend.openConversation`: start the MCP
      server in the existing supervised scope, write a temp
      `.mcp.json` in `workDir`, pass `--mcp-config` (and
      `--allowedTools ask_user`) into the args. Tear down on
      conversation close.

- [ ] **M5.** `ClaudeConversation` consumes the bridge. A fork
      inside the conversation drains questions, emits
      `ConversationEvent.UserQuestion(text, answer => bridge.reply(answer))`,
      lets the renderer prompt, signals the bridge with the typed
      answer. `canAskUser = true`.

- [ ] **M6.** System-prompt nudge. Inject one line into
      `--append-system-prompt-file` (or prepend to the user prompt)
      telling the agent it has an `ask_user` tool for clarifying
      questions. Only applies on the interactive path — autonomous
      callers don't get the MCP server or the nudge.

- [ ] **M7.** Tests: AskUserBridge concurrency, MCP server
      `tools/list` smoke, ClaudeConversation+bridge integration.
      End-to-end remains manual.

- [ ] **M8.** ADR 0012 — MCP host bridge. Architecture, why chimp,
      lifecycle, why HTTP over stdio, future generalisation (other
      host-side tools).
